# Security policy

## Reporting a vulnerability

**Do not open a public issue.** Use GitHub's private reporting:

1. Go to [Security → Report a vulnerability](https://github.com/jeloercc/pqc-sdk/security/advisories/new).
2. Describe the problem: affected version, impact, and steps or a PoC to
   reproduce it.

What to expect:

- **Acknowledgment within 72 hours.**
- Assessment and a fix plan (or a justified dismissal) within 14 days.
- Coordinated disclosure: we publish the advisory and credit the reporter
  (if you want it) together with the patched version.

## Supported versions

| Version      | Support                         |
| ------------ | ------------------------------- |
| Latest `0.x` | ✅ Security fixes               |
| Older        | ❌ Update to the latest version |

While the project is on `0.x`, security fixes land only on the latest
published version of `@pqc-sdk/core` and `@pqc-sdk/cli`.

## Scope

**In scope:** the code in this repo — the `@pqc-sdk/core` API (hybrid
encryption, signatures, key serialization), the CLI, and any incorrect use we
make of the primitives (nonces, derivation, handling of sensitive material).

**Out of scope (report upstream):** the primitives themselves. ML-KEM and
ML-DSA come from [`@noble/post-quantum`](https://github.com/paulmillr/noble-post-quantum)
and AES-256-GCM from [`@noble/ciphers`](https://github.com/paulmillr/noble-ciphers);
vulnerabilities in those implementations should be reported to their
maintainers.

## Threat model: known limits

So you can assess whether the SDK fits your case:

- `@noble/post-quantum` has no independent audit yet (self-audit 04/2026).
- As with any JavaScript implementation, there are no strict constant-time
  execution guarantees; local side-channel attacks are outside the threat
  model.
- **No memory zeroization.** Shared secrets (the AES-256 key), decrypted
  plaintext, and secret-key bytes are not wiped after use. JavaScript offers no
  reliable zeroization primitive and `@noble` does not zeroize either, so this
  is an ecosystem limitation, not something the SDK can fully close.
- The keys generated by `pqc init` are development keys and the CLI warns
  about it: do not use them in production.

## Dependency policy

The cryptographic dependencies (`@noble/post-quantum`, `@noble/ciphers`) are
pinned to **exact versions** in `@pqc-sdk/core`, not caret ranges. A downstream
`npm install` therefore resolves the exact primitive build that shipped and
passed the NIST vectors; bumps are made deliberately behind the vector tests.
Published tarballs include source maps (`*.js.map` / `*.cjs.map`) — expected for
this MIT project, noted here so it is a conscious choice.

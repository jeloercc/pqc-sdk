# Pre-launch audit — @pqc-sdk (June 2026)

> **Scope & stance.** This is a skeptical, external-style pre-launch review of
> `@pqc-sdk/core` and `@pqc-sdk/cli` at `0.2.0`, covering crypto correctness,
> API safety, CLI security, supply chain, docs accuracy, and edge-case test
> coverage. It is a findings report only — **nothing was changed**. It is not a
> formal cryptanalysis or an independent professional audit, and it does not
> review the upstream `@noble/*` implementations.
>
> **Method / evidence collected this run:** full read of `packages/core/src/**`
> and `packages/cli/src/**`; `tsc` assignability check of the published
> `deserialize` example; `pnpm test` (64 passed, 94.3% lines / 92.1% branches);
> `pnpm audit --prod` (clean); `npm pack --dry-run` for both packages; review of
> all three GitHub workflows; `.gitignore` and `git ls-files` key-leak check.

## Headline

The cryptographic core is sound. The KEM-DEM construction, implicit-rejection
handling, nonce strategy, AAD header binding, and key validation are all
correct (see "Confirmed sound" below). **No Critical or High findings.** The
most material items are operational/DX: the CLI generates secret-key files with
no `.gitignore` protection (secret-commit risk), and two published docs claims
are inaccurate (one says the SDK implements SLH-DSA; one reference example does
not compile).

## Medium

### [Medium][CLI/secrets] M1 — Secret-key files are generated with no `.gitignore` protection

**Where:** `packages/cli/src/keyfiles.ts:42-66`, `packages/cli/src/commands/init.ts:45-61`, root `.gitignore` (no `keys/` or `*.pqc` entry).

The headline onboarding flow is `npx @pqc-sdk/cli init`, which writes a real
ML-KEM secret to `keys/dev.secret.pqc`. `keygen` does the same for
user-chosen names/dirs. The secret is correctly written with mode `0600`
(`keyfiles.ts:62-63`), but neither command creates or updates a `.gitignore`,
and the root ignore file lists no key pattern. A user who runs `git add .`
commits their secret key. For a security SDK this is the most likely real-world
footgun. `init`'s keys are verbally marked development-only (`init.ts:59`), which
softens it, but `keygen` carries no such framing and is the path users will use
for keys they treat as real.

**Suggested fix:** Have `init` create/append a `.gitignore` containing `keys/`
and `*.secret.pqc`; have `keygen` print a one-line reminder and, when a
`.gitignore` exists, offer to add the pattern. Cheap, removes the footgun.

### [Medium][Docs] M2 — Docs claim the SDK implements FIPS 205 / SLH-DSA

**Where:** `apps/docs/guide/why-pqc.md:16-17` (also `CLAUDE.md` "Algorithms" line lists SLH-DSA).

> "FIPS 203 (ML-KEM) … FIPS 204 (ML-DSA) and **FIPS 205 (SLH-DSA)** for
> signatures. **Those are the algorithms this SDK implements.**"

`SUPPORTED_ALGORITHMS` is `['ml-kem-768', 'ml-dsa-65']` (`packages/core/src/index.ts:45`);
there is no SLH-DSA anywhere in `core`. This is the same class of inaccuracy the
internal review caught for the npm description (its finding #1) — it survived in
the docs site. Overstating implemented algorithms is exactly what a skeptical
buyer checks first.

**Suggested fix:** Reword to distinguish "NIST standardized 203/204/205" from
"this SDK implements 203 and 204; SLH-DSA is not implemented." Align `CLAUDE.md`
to mark SLH-DSA as a roadmap goal, not a current algorithm.

### [Medium][Docs/API] M3 — The published `deserialize()` example does not compile

**Where:** `packages/core/src/keys.ts:79-91` (JSDoc) → rendered in `apps/docs/api/functions/deserialize.md:23-28`.

The canonical example is:

```ts
const publicKey = pqc.keys.deserialize(tokenReceivedFromClient);
const ciphertext = await pqc.encrypt(payload, publicKey);
```

`deserialize` returns the wide `PqcKey<Algorithm, KeyUse>`, which is **not**
assignable to `encrypt`'s `PublicKey<'ml-kem-768'>`. Verified this run with the
package's own tsconfig:

```
src/_tccheck.ts(6,47): error TS2345: Argument of type 'PqcKey<Algorithm, KeyUse>'
  is not assignable to parameter of type 'PublicKey<"ml-kem-768">'.
```

So the reference example for a core function fails to typecheck — and `CLAUDE.md`
requires every public function to ship a _working_ JSDoc example. The guide
pages get this right (`apps/docs/guide/encrypt-files.md:26,49` use an explicit
`as never` and explain it); only the canonical API example is wrong. Because it
lives in source JSDoc, it is not twoslash-checked at docs build time, so it
ships broken.

**Suggested fix:** Mirror the guide's `as never` (and the tip explaining it) in
the JSDoc, or better, ship a narrowing helper / typed overload (see L1) and use
it in the example so copy-paste compiles.

## Low

### [Low][API safety] L1 — Wide `deserialize` return type erodes compile-time guarantees

**Where:** `packages/core/src/keys.ts:92`; usage pattern at `apps/docs/guide/encrypt-files.md:26,49`, `packages/core/src/nist-vectors.test.ts:51-55`.

`deserialize` returns `PqcKey`, so every real call into `encrypt`/`decrypt`/
`sign`/`verify` requires an `as never` (or similar) cast, which the docs
themselves teach. This does **not** create silent weak crypto — `requireKey`
(`algorithms.ts:67-90`) re-validates algorithm, use, and length at runtime, so a
mis-cast key fails closed with `WRONG_ALGORITHM` / `WRONG_KEY_USE` / `INVALID_KEY`.
But normalizing `as never` in user code defeats the otherwise-good compile-time
type discrimination and invites genuine cast mistakes elsewhere.

**Suggested fix:** Add typed entry points — e.g. `deserialize<A, U>()` overloads
that validate against expected algorithm/use, or `deserializePublicKey('ml-kem-768', token)`
helpers that return the narrow type — so users never reach for `as never`.

### [Low][Supply chain] L2 — Cryptographic deps float on caret ranges in the published package

**Where:** `packages/core/package.json` (`@noble/post-quantum: ^0.6.1`, `@noble/ciphers: ^2.2.0`).

`pnpm audit --prod` is clean and the repo lockfile pins for CI, but the
**published** package ships caret ranges. Downstream `npm install` can therefore
resolve a newer, untested minor/patch of the actual cryptographic primitives
than what shipped and passed the NIST vectors. For a crypto SDK, the blast radius
of an unexpected primitive bump (or a compromised release) is higher than for a
typical library. npm provenance (correctly configured, see "Confirmed sound")
protects _your_ publish, not the consumer's transitive resolution.

**Suggested fix:** Pin the two `@noble/*` cryptographic dependencies to exact
versions (or a tight range), bump them deliberately behind the vector tests, and
note the policy in `SECURITY.md`.

### [Low][Crypto] L3 — `decapsulate` runs outside `decrypt`'s try/catch

**Where:** `packages/core/src/encrypt.ts:103-111`.

```ts
const sharedSecret = spec.kem.decapsulate(kemCiphertext, secretKey.bytes); // outside try
try {
  return Promise.resolve(gcm(sharedSecret, nonce, header).decrypt(sealed));
} catch { throw new PqcError('DECRYPTION_FAILED', …); }
```

For a valid-length secret key ML-KEM uses implicit rejection and does not throw,
so in practice this is benign. But any future/edge case where `decapsulate`
throws would surface a raw `@noble` error instead of a normalized `PqcError`,
breaking the documented contract that decryption failures are always
`DECRYPTION_FAILED`. Defense-in-depth.

**Suggested fix:** Move `decapsulate` inside the `try`, normalizing any throw to
`DECRYPTION_FAILED` (or `INVALID_CIPHERTEXT`).

### [Low][API/crypto] L4 — Signature `context` length is not validated

**Where:** `packages/core/src/sign.ts:26-33,48-62`; `packages/core/src/types.ts:34-37`.

FIPS 204 caps the context string at 255 bytes. The SDK passes `options.context`
straight through to `@noble` without validating length. `@noble` will reject an
oversized context by throwing — but in `sign` that throw escapes as a raw error
(not a `PqcError`), while `verify` swallows it via its `catch` and returns
`false`. Inconsistent and undocumented surface for the same invalid input.

**Suggested fix:** Validate `context.length <= 255` up front and throw a
`PqcError` consistently from both `sign` and `verify`; add a test.

### [Low][Docs/governance] L5 — Spanish comments in CI workflows violate the English-only rule

**Where:** `.github/workflows/release.yml` (`# crear el PR de versiones…`, `# OIDC para npm provenance`), `.github/workflows/docs.yml` (`# Un solo deploy a la vez…`).

`CLAUDE.md` mandates that all user-facing text — explicitly including code
comments — be English. These workflow comments are the only remaining
non-English text found.

**Suggested fix:** Translate the three comments.

### [Low][CLI] L6 — `init` overwrites `example.ts` unconditionally

**Where:** `packages/cli/src/commands/init.ts:46-52`.

`init` guards on `pqc.config.json` existence (`init.ts:46-48`) and `writeKeyPair`
refuses to clobber existing keys, but `writeFile('example.ts', EXAMPLE)`
(`init.ts:52`) runs unconditionally and would overwrite a user's existing
`example.ts`. Low impact (only fires in an un-initialized project) but
inconsistent with the otherwise careful no-clobber posture.

**Suggested fix:** Skip (or prompt) when `example.ts` already exists.

## Informational

### [Info][Tests] I1 — Edge-case coverage gaps

Coverage is 94.3% lines and all 64 tests pass, but the following items from the
requested edge-case checklist are **not** exercised:

- **Multi-MB payloads:** the largest encrypt/decrypt test is 100 KB
  (`encrypt.test.ts:37`), yet docs claim correctness "1 byte or 100 MB"
  (`hybrid-encryption.md:46`). No multi-MB roundtrip test.
- **Invalid base64url length:** the `length % 4 === 1` "impossible length"
  branch is uncovered (`base64url.ts:22-24`), and no `deserialize` test feeds a
  token of that length.
- **Right algorithm+use, wrong byte length at an operation entry point:**
  `requireKey`'s length guard (`algorithms.ts:83-88`) is uncovered — e.g. a
  hand-built or truncated-in-memory key passed to `encrypt`/`sign` is untested
  (the deserialize path validates length separately).
- **`serialize` length guard** (`keys.ts:74-75`) and **`generateKeyPairFromSeed`
  seed-length guard** (`keys.ts:46-50`) are uncovered.
- **`verify` catch path** (`sign.ts:60-61`) is never hit — no input forces
  `@noble.verify` to throw rather than return `false`.
- **Oversized signature context** (>255 B) — untested (see L4).
- **Empty-message signing** — untested (encrypt has an empty-input test; sign
  does not).

Covered well: tamper detection across header/KEM/nonce/sealed regions, wrong-key
and wrong-algorithm rejection, truncated ciphertext, invalid base64url
characters, non-canonical trailing bits, and the NIST ACVP vectors (31 cases).

### [Info][Crypto] I2 — No zeroization of sensitive buffers

`sharedSecret` (the AES-256 key), decrypted plaintext, and secret-key bytes are
not wiped after use (`encrypt.ts`, `keys.ts`). JavaScript provides no reliable
memory zeroization and `@noble` does not zeroize either, so this is a known
ecosystem limitation rather than a defect — noted for completeness and worth a
sentence in `SECURITY.md`'s threat model.

### [Info][CLI] I3 — `--out` accepts arbitrary directories; `--name` traversal guard is partial

**Where:** `packages/cli/src/keyfiles.ts:28-39`, `packages/cli/src/commands/keygen.ts:35`.

`assertSafeName` correctly blocks `/`, `\`, and `..` in `--name`, but `--out`
takes any directory (including `../..` or absolute paths). This is by design (the
user is writing their own keys on their own machine), but it means the `..`
restriction on `--name` is partly cosmetic — `--out ../elsewhere` already escapes
the working directory. No action required; documented so reviewers don't read the
name guard as a sandbox.

### [Info][Supply chain] I4 — Published tarballs ship source maps

`npm pack --dry-run` shows both packages ship `*.js.map` (and `*.cjs.map` for
core), which embed original source paths/content. Harmless for an MIT
open-source project; flagged only so it's a conscious choice. Tarballs are
otherwise clean: `dist/`, `LICENSE`, `README.md`, `package.json` — no source,
tests, keys, or `.env`.

## Confirmed sound (checked, no finding)

- **KEM-DEM construction:** a fresh ML-KEM-768 encapsulation per message yields a
  unique 32-byte shared secret used directly as the AES-256-GCM key
  (`encrypt.ts:38-44`). Fresh key per message means the random 96-bit nonce
  (`encrypt.ts:39`) never collides under a reused key — nonce uniqueness is not a
  concern, and there is no API to inject a caller-supplied key or nonce.
- **Implicit rejection:** ML-KEM's pseudo-random decapsulation output on bad
  input is handled correctly — integrity rests on the GCM tag, and every
  decryption failure funnels to a single unambiguous `DECRYPTION_FAILED`
  (`encrypt.ts:104-111`). No padding-oracle / distinguishable-failure surface.
- **AAD header binding:** the 2-byte `(version, algId)` header is bound as
  AES-GCM AAD and reconstructed from the received bytes on decrypt
  (`encrypt.ts:43-44,93-95`), with a fail-fast pre-check that is correctly
  documented as _not_ a substitute for the tag (`encrypt.ts:80-91`).
- **Direct use of the ML-KEM secret as the AES key** is appropriate (FIPS 203
  output is uniform; no KDF required), and is accurately documented
  (`hybrid-encryption.md:39-41`).
- **No key material in error messages or stack traces:** error strings carry
  only lengths, algorithm names, and key _use_ — never raw bytes
  (`algorithms.ts:75-88`, `keys.ts`, `errors.ts`). base64url errors echo at most
  one offending character of already-invalid encoded input.
- **Compile-time misuse prevention works:** passing a secret where a public key
  is expected, or an ML-DSA key to `encrypt`, is a type error
  (`encrypt.test.ts:48,57`, `sign.test.ts:64,75`), and runtime `requireKey`
  backstops it. No path lets a user silently downgrade to weak crypto (only one
  KEM, fixed AES-256-GCM, no algorithm negotiation).
- **CLI key files** are written `0600` with an explicit `chmod`
  (`keyfiles.ts:62-63`), so umask cannot widen them; dev keys are clearly framed
  (`init.ts:59`).
- **`audit` never executes or evals scanned code** — it only runs `RegExp.test`
  on each line (`audit.ts:123-129`) and reads files behind a 1 MiB cap
  (`audit.ts:117-121`). No code execution, with explicit heuristic framing.
- **Supply chain:** `pnpm audit --prod` clean; npm provenance correctly wired
  (`release.yml`: `id-token: write` + `NPM_CONFIG_PROVENANCE: 'true'`); version
  strings injected from `package.json` at build time, not hardcoded.
- **NIST ACVP vectors** (FIPS 203 keygen + encap/decap, FIPS 204 keygen + sigVer)
  run in CI and pass (31 cases) — substantive validation, not vacuous.

## Suggested remediation order

1. **M1** (gitignore for secret keys) — smallest change, biggest real-world risk reduction.
2. **M2, M3** (SLH-DSA claim, broken `deserialize` example) — accuracy before launch.
3. **L1, L4** (typed `deserialize`, context validation) — close the API-safety ergonomics.
4. **L2** (pin crypto deps), **L3, L5, L6**, then the **I1** test gaps.
